# 前端规划与需求文档（评审草案）

> 项目：job_info_collector
> 文档版本：v0.1（待评审）
> 日期：2026-03-15
> 状态：Draft

## 1. 文档目标

本文件用于在前端开发前统一目标、范围、优先级、验收标准与交付节奏，作为后续实现、联调、发布和回归测试的基线。

本文件聚焦“静态前端展示层”，不改动现有爬取层和分析层的核心职责边界。

## 2. 背景与现状

### 2.1 业务背景

当前项目已形成多站点岗位聚合能力，数据经过统一字段契约输出，具备前端展示价值。前端需要承担：

- 岗位信息可浏览（卡片化）
- 多维筛选联动
- 多字段关键词检索
- 渐进式加载和弱网可用

### 2.2 已有共识（来自项目规范）

- 前端数据读取路径应基于：
  - web/data/jobs.index.json
  - web/data/chunks/jobs-*.json
  - web/data/jobs.json（回退）
- 首屏不应依赖大包全量数据，优先 index + 首批 chunks。
- 邀请码通过后，加载流程必须继续推进到可见列表状态。
- 关键交互链路：筛选、搜索、分页加载必须可回归验证。

### 2.3 当前缺口（待补齐）

- 缺少一份结构化前端需求基线（功能边界、验收口径、里程碑）。
- 缺少明确的前端信息架构和状态机约束文档。
- 缺少统一的前端测试清单（功能 + 性能 + 发布前检查）。

## 3. 目标与非目标

### 3.1 目标（MVP）

1. 在静态站点场景下，稳定展示岗位列表。
2. 支持联动筛选（公司、项目、类别、城市）和关键词搜索。
3. 支持 index + chunks 渐进加载，并具备 jobs.json 回退。
4. 在移动端和桌面端均可用。
5. 发布到 GitHub Pages 后可稳定访问。

### 3.2 非目标（本阶段不做）

1. 用户登录体系与个性化推荐。
2. 后端在线检索服务（HTTP API 实时检索）。
3. 收藏投递、账号同步等长期能力。
4. 复杂可视化仪表盘（留到分析联动阶段）。

## 4. 用户与核心场景

### 4.1 目标用户

- 校招求职者：快速找到匹配岗位。
- 数据维护者：验证新站点数据能否在前端正确消费。
- 项目协作者：通过页面快速感知数据质量与覆盖范围。

### 4.2 核心任务场景

1. 按城市 + 类别筛选岗位。
2. 输入关键词检索职责/要求/标题。
3. 快速查看岗位详情入口并跳转原始链接。
4. 在弱网下先看到首屏，再逐步补齐后续数据。

## 5. 功能需求

### 5.1 数据加载与容错

1. 启动时先定位 data 目录。
2. 优先请求 jobs.index.json。
3. 根据 index 的 chunks 列表并发拉取首批分片。
4. 首批渲染完成后后台补齐剩余分片。
5. 任一分片失败时可重试；全部失败时回退 jobs.json。
6. jobs.json 不可用时，显示可操作错误提示（含排查建议）。

### 5.2 列表展示

1. 卡片字段至少包含：公司、岗位名、城市、类别、发布时间、来源链接。
2. 空字段按占位规则显示（不打断布局）。
3. 列表初始展示数量可配置，支持“加载更多”。
4. 对超长文本做折叠/截断，并提供展开能力（移动端优先可读）。

### 5.3 筛选与搜索

1. 筛选维度：公司、项目、类别、城市（MVP 必须）。
2. 筛选需联动：上游条件变化后，下游可选项即时收敛。
3. 搜索范围：标题、职责、要求、加分项。
4. 搜索输入防抖，避免高频重算。
5. 筛选与搜索组合后结果总数实时更新。

### 5.4 交互与状态

1. 必须定义并实现 loadState：准备中 -> 定位数据目录 -> 加载中 -> 完成/兼容模式/失败。
2. 邀请码验证通过后必须解除遮罩，且不得阻断初始化。
3. 空结果页需区分“无匹配结果”和“加载失败”。
4. 关键操作保留可见反馈（加载中、重试中、失败原因）。

### 5.5 兼容性与发布

1. 支持本地静态服务启动调试（http.server）。
2. 支持 GitHub Pages 子路径部署。
3. 禁止依赖 file:// 直接打开页面。

## 6. 非功能需求

### 6.1 性能

1. 首屏目标：3 秒内看到首批岗位（常规网络）。
2. 筛选/搜索交互响应：200ms 到 300ms 内可见反馈。
3. 避免一次性加载全量大包造成主线程阻塞。
4. 移动端降级策略：跳过非关键预计算。

### 6.2 可维护性

1. 数据加载、筛选搜索、渲染逻辑模块化拆分。
2. 新增筛选维度时不允许硬编码多处散落逻辑。
3. 关键常量（分页大小、防抖时间）集中配置。

### 6.3 可观测性

1. 控制台日志分级：info/warn/error。
2. 对加载链路关键节点打点日志（index 成功、chunk 失败、fallback 命中）。
3. 发布前保留最小调试信息，避免过量噪音。

### 6.4 可用性

1. 桌面与移动端可用。
2. 交互控件点击区域满足移动端操作。
3. 文案明确、错误可理解。

## 7. 信息架构与页面草图（文本版）

1. 顶部区：标题、更新时间、数据来源说明。
2. 工具区：关键词搜索、筛选器、清空条件。
3. 状态区：当前数据状态、匹配数量、加载反馈。
4. 列表区：岗位卡片列表。
5. 底部区：加载更多、版权/免责声明。

## 8. 数据契约与前端映射

### 8.1 输入字段（来自统一契约）

至少兼容：

- company
- job_id
- title
- recruit_type
- job_category
- job_function
- work_city
- work_cities
- responsibilities
- requirements
- bonus_points
- tags
- publish_time
- detail_url
- fetched_at
- source_page

### 8.2 映射规则

1. work_cities、tags 统一转数组后渲染。
2. publish_time 优先标准化为本地可读时间。
3. 缺失字段显示为空占位，不允许抛异常中断渲染。
4. 唯一标识优先 company + job_id，缺失时回退 detail_url。

## 9. 验收标准

### 9.1 功能验收

1. 页面可加载并展示岗位列表。
2. 筛选联动正确，搜索有效。
3. 邀请码通过后能继续加载并展示数据。
4. index + chunks 路径可工作，jobs.json 回退可工作。

### 9.2 性能验收

1. 首屏可见时间满足目标。
2. 高频筛选/搜索无明显卡顿。

### 9.3 发布验收

1. 已生成 jobs.index.json、jobs.json、chunks。
2. GitHub Pages 环境首屏不请求 jobs.json 大包（正常路径下）。
3. 页面显示数据更新时间与来源。

## 10. 实施计划（建议）

### 阶段 A：需求冻结与基线搭建（1-2 天）

1. 确认本文件评审意见并冻结 MVP。
2. 确认目录结构与构建/部署方式。
3. 建立最小页面骨架和加载状态机。

### 阶段 B：核心功能实现（2-4 天）

1. 完成数据加载链路（index/chunks/fallback）。
2. 完成卡片渲染、筛选联动、搜索、防抖。
3. 完成移动端适配与“加载更多”。

### 阶段 C：联调与发布准备（1-2 天）

1. 对接数据导出产物做联调。
2. 跑回归清单并修复高优先级问题。
3. 完成发布前检查并部署。

## 11. 风险与对策

1. 风险：分片文件缺失或路径错误导致列表为空。
- 对策：保留 jobs.json 回退；错误提示带排查指引。

2. 风险：邀请码流程阻断初始化。
- 对策：将遮罩控制与初始化解耦，增加状态断言。

3. 风险：新增筛选维度破坏联动逻辑。
- 对策：统一筛选状态管理，增加回归用例。

4. 风险：移动端性能抖动。
- 对策：降低预计算、分批渲染、减少同步重算。

## 14. 前端专属智能体维护策略（执行版）

后续前端相关改动默认由一个专属智能体负责，避免多人并行修改导致风格漂移和状态机回归。

### 14.1 维护边界

专属智能体可直接维护以下内容：

1. `web/index.html`
2. `web/assets/styles.css`
3. `web/assets/app.js`
4. `web/config/invite-codes.js`
5. 与前端静态数据契约直接相关的导出脚本（`src/export_frontend_jobs.py`）
6. 前端发布校验相关工作流片段（如 pages 部署校验步骤）

### 14.2 变更准入规则

每次前端改动必须同时满足：

1. 不破坏 `index + chunks + jobs.json fallback` 数据加载链路。
2. 不破坏邀请码验证通过后的初始化流转（loadState 必须继续推进）。
3. 不破坏四维联动筛选（公司/项目/类别/城市）与排序能力。
4. 保持移动端可用（筛选、卡片、加载更多可操作）。
5. 修改后同步更新本文件或 `docs/DECISIONS.md` 的关键决策。

### 14.3 专属智能体标准作业流（SOP）

1. 先读本文件第 5、6、9、12、14 节再动代码。
2. 先做最小改动，避免无关重构。
3. 代码修改后至少完成：
- 关键文件静态检查（语法/引用错误）
- 数据加载链路自检（首批、补齐、fallback）
- 移动端布局快速检查
4. 输出变更摘要时必须说明：
- 改了哪些文件
- 是否影响数据契约
- 是否影响回归清单

### 14.4 升级与回滚约束

1. 若前端改动涉及数据字段新增/删减，必须先更新契约说明再改代码。
2. 若上线后出现移动端加载回退到固定条数等问题，优先回滚到最近可用前端提交，再做增量修复。
3. 禁止在一次提交中同时引入大规模视觉重构与数据加载核心重构。
